# MFlux Studio — 迁移与实施计划

## 1. 概述

本文档描述 MFlux Studio 从当前单体本地原型到更加健壮、可维护版本的分阶段迁移计划。

## 2. 当前架构快照

### 2.1 现状

- **部署**：单体本地，前后端在同一主机
- **数据库**：SQLite，Alembic 依赖已安装但未初始化
- **任务队列**：内存 deque（进程重启后丢失）
- **鉴权**：无
- **日志**：console + rotating file
- **测试**：Playwright E2E（49 项，全部通过）
- **文档**：README、AGENTS.md、BUG_REPORT.md（已产出）

### 2.2 已知问题

| 问题 | 影响 | 当前状态 |
|------|------|----------|
| 模型删除仅操作数据库记录 | 物理文件残留 | 需改进 |
| 任务取消无法停止正在运行的子进程 | 不能真正停止 mflux | 需改进 |
| 无 CORS 管控（`allow_origins=["*"]`） | 网络暴露时存在风险 | 注意项 |
| image_path 存储 URL 而非文件路径 | 已修复（Bug 3） | ✅ |
| FLUX.2 不支持 negative_prompt | 已修复（Bug 1） | ✅ |
| SQLite 单线程写限制 | 多用户场景受限 | 注意项 |

## 5. 风险与缓解措施

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| SQLite 并发写入冲突 | 低 | 中（数据丢失） | 监控写入频率，必要时切换 PostgreSQL |
| mflux CLI 版本不兼容 | 中 | 高（生成失败） | 在 Generator 中添加版本检测和兼容性检查 |
| 子进程内存泄漏 | 中 | 中（系统变慢） | 任务超时后强制 kill，定期监控内存 |
| 磁盘空间耗尽 | 低 | 高（生成失败） | 已实现 `_check_disk_space`，增加定期清理 |
| 前端代理配置变更 | 低 | 中（API 不可达） | Vite proxy 配置纳入版本控制 |
| 模型扫描性能 | 低 | 中（扫描耗时） | 已实现 HuggingFace cache 缓存路径识别 |

## 7. 兼容性与回滚策略

- **数据库迁移**：每次 Alembic 迁移必须有对应的 `downgrade()` 方法
- **API 版本**：重大变更时使用 URL prefix `/api/v2`，兼容旧版本
- **前端回滚**：所有静态文件通过 Vite build 产物部署，保留上一版本备份
- **配置回滚**：`settings` 表中的每项设置单独存储，可逐项恢复
